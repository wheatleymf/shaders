Your First Unlit Shader

IN THIS ARTICLE: Understanding the anatomy of s&box shaders, figuring out vertex and pixel structs, how do floats work, creating first unlit shader and learning where does data come from.

If you want to skip whole "figuring out the shader anatomy" part, you can use s&box editor to create a new shader file using a preset. If you want to learn more about the file anatomy step by step, feel free to follow this part.

It is time to create our first shader! Lets start with something very simple: an unlit shader. What is an unlit shader? This usually implies a shader that isn't affected by light or shading. It returns color with exact colors that shader had.

Creating a new file

Lets create a new blank text file. Give it a .shader file extension, and throw it shaders folder inside of your s&box project. At the moment of writing this article you can create a new template shader right in s&box, but for the sake of a better article's flow and explaining every shader bit step by step, lets create it from scratch.

All of your .shader files are compiled by s&box automatically. Whenever there's an update in your shader file, s&box will catch and attempt to compile it. The only exception are custom .hlsl files referenced in your shader as an #include, but they are optional and we don't need to make them right now.

Shader compiler notifications are logged in your s&box editor console. If there are any errors, they will be displayed there.

Sections & Shader Anatomy

All shaders have a specific internal structure, and each has it's own purpose. Let me show how this structure generally looks like in an average shader. Don't copy it just yet, lets implement each section by ourselves. 


// Shader structure, as of November 2024

// FEATURES block allows implementing options that you can click in left bar
// of the Material Editor. Features can be used for controlling static combos.
FEATURES
{
    // ...
}

// MODES implements shader modes. They define ways your shader will render
// under different passes or conditions. 
// https://docs.facepunch.com/s/sbox-dev/doc/modes-SFu3UxPnhm
MODES 
{
    // This is a main rendering mode. You may want to always include it. 
    VrForward();
    // ...
}

// COMMON block allows defining variables and other data that you can access
// from anywhere, like from both vertex and fragment shader. 
COMMON
{
    // ...
}

// In this area you can define any structs, or other stuff. 
// You may want to define vertex and pixel input structs here.
...

// VS is a Vertex Shader. This block is your vertex shader.
// Main function of vertex shader is MainVs().
VS
{
    // ...
}

// PS is a Pixel Shader (fragment shader).
// This is a final shader stage and that's where you manipulate the color. 
// Main function of fragment shader is MainPs(), and it returns float4.
PS
{
    // ...
}
If you looked up the code from some older s&box shaders, you probably noticed that they have a HEADER block right at the top. It is not necessary for your shader, and as of now, none of the information entered there is visible inside the engine. Feel free to ignore it.

FEATURES block

FEATURES allows implementing special options for your shader that you can disable/enable via Material Editor in the left bar. Generally features are needed for defining special shader variants, features variable value are used by static combos. We'll mess around with static combos and see how are they working later.

For now, lets implement the bare minimum. We need to include a core HLSL file which contains all built-in options, without these our shader won't compile, for a whatever reason. Add this into your new shader file: 


FEATURES
{
    #include "common/features.hlsl"
}

If you're wondering what are we including, this #include will add features like "Do not cast shadows", "Render backfaces", as well as some other core features. Texture Filtering and Additive Blend won't work out of the box, which is awkward.

MODES block

Lets add MODES block. This block defines how our shader is handled in different passes. For now, we need only VrForward(). You can see all available modes on s&box documentation page here.

  • VrForward() is a standard rendering mode. This is where all primary rendering happens, your shader won't compile without this mode.
  • Depth() will make shader write to depth. This is necessary for shadow rendering and few other render things like ambient occlusion, and anything else what relies on depth map.
  • ToolsVis( S_MODE_TOOLS_VIS ) is DEPRECATED, but you may see it in some older shader files. As of now, it's been deprecated and you don't have to include it anymore. I'll mentioned it here, just in case. 


MODES
{
	VrForward();
    Depth();
}
Depth() will be replaced with DepthNormal() in the near future. I don't know if old shaders will continue working, but you'll definitely need to replace it with a new method when it's out. It should bring support for G-buffer, which is a nice addition.

COMMON block

Next up is a COMMON block. All data and variables defined here will be accessible from any shader stage. By default, if you declare a variable (or add a texture input) inside the vertex shader, you won't be able to access it from a fragment shader. But if you add texture input/variable definitions inside the COMMON block, you will be able to access them from both vertex and fragment shader.

Since you don't need to add anything that requires access from multiple shader stage yet, lets just include a core HLSL file that automatically imports all necessary stuff to get our shader compiling and move forward. 


COMMON
{
	#include "common/shared.hlsl"
}

Structs

Vertex Shader Input Struct

Before you can use it to build and render your mesh with this new shader, this data must be sorted into an appropriate struct, with correctly set up semantics. Luckily, s&box comes with pre-built vertex and pixel input structs, which means we don't have to set up any of these by ourselves. (you can always do it from scratch if you want to, but for now lets stick with what s&box offers to us!)

  • What is inside the vertex shader input struct? Mesh vertices are represented by following data: object space vertex position, texture coordinates, and object space normals. Built-in VertexInput also comes with some additional stuff, that can help with instancing, skinning and a few other optional features.
  • Why do you need it? This data is required by vertex shader to perform a number of operations, like transforming vertices into the projection space, calculating mesh tangents, geometrical normals, and fill out other data that will be later utilized in fragment (pixel) shader.

To add built-in VertexInput struct, just do this: 


struct VertexInput
{
	#include "common/vertexinput.hlsl"
};

Perfect! When you reference this struct in vertex shader, it'll automatically read the struct data from included .hlsl file. If you want to read more about vertex and pixel structs, s&box docs have a dedicated page for them.

Pixel Shader Input Struct

Let's do the same thing as with vertex struct. Use the built-in struct, it will be more than enough for us. When vertex shader finishes transforming and manipulating all vertex things, it returns a PixelInput object. This object is then used by fragment shader as a source of data to work with. This is where you are going to grab all things from: texture coordinates, pixel position, viewport UV, mesh geometry normals, etc..

Both structs are utilized by s&box default shading model, so nearly all s&box shaders you will find online are utilizing these two default structs. This is pretty convenient as it contains almost everything you need. 


struct PixelInput
{
	#include "common/pixelinput.hlsl"
};

And now all base data is included! Next step is adding a vertex shader.

Vertex Shader

As I've mentioned a few times in this article, vertex shader grabs the structured mesh data and does bunch of manipulations. Generally, this is where you may want to calculate geometry normals, and transform vertex position into the projection space. Optionally, this is where you can also manipulate vertex positions, which will affect the shape of a mesh. When it's all done, vertex shader returns an object with all transformed data that is later used by fragment shader.

Processing Vertices

s&box has two main vertex helper functions that handle all these routine operations automatically, which means you don't need to worry about instancing and setting up projection space yourself. These functions are:

  • ProcessVertex( VertexInput i ) - grabs the vertex shader input data and does most basic stuff for you: set up instancing and skinning, and then calculate vertex position in world space with geometry normals and tangents. This function takes VertexInput vertex and returns a new vertex with a type of PixelInput.

    You can use data from object this function returns to do any sorts of manipulations: vertex displacement, normal/UV manipulations, and anything else you'd like.
  • FinalizeVertex( PixelInput o ) - before returning the vertex, it's position must be converted into the projection space. This is what FinalizeVertex does.

These vertex helpes are documented on the s&box documentation website

Implementing the Vertex Shader

Okay, at first of all, create a new block like with previous sections: add VS and then add brackets on the next line. Inside this block, we need to include another core file, which is common/vertex.hlsl. Your section currently should look like this: 


VS
{
    #include "common/vertex.hlsl"

    // ...
}

Next step is adding the main function of a vertex shader. Main function in vertex shader is called MainVs( input ), where input is a vertex input struct you have just defined. In your case, it's VertexInput. Vertex shader must return a finalized vertex with a type of PixelInput that will be used by a fragment shader. 


VS
{
    #include "common/vertex.hlsl"

    // "MainVs" is the name of a main vertex shader function. It always must be MainVs.
    // VertexInput is the "unprocessed" vertex data that we're feeding into vertex shader. It is our input. 
    // PixelInput is a type that vertex shader will return. Object of this type will be sent to fragment shader.
    PixelInput MainVs( VertexInput i )
    {
        // ...
    }
}

Inside the brackets we can start doing all sorts of magic with the given vertex. Lets keep it minimal: process and finalize the vertex using helper functions, and then return it.

Declare a new variable with PixelInput type that is initialized by ProcessVertex( i ). Give it any name - it doesn't matter how is it called. Most shaders keep the name very minimal, something like o.

And now, since we don't plan editing anything in this vertex data, just return FinalizeVertex( o ). Your vertex shader is now ready! It will return a PixelInput object that you will utilize in fragment shader. 


        VS
        {
            #include "common/vertex.hlsl"
        
            // "MainVs" is the name of a main vertex shader function. It always must be MainVs.
            // VertexInput is the "unprocessed" vertex data that we're feeding into vertex shader. It is our input. 
            // PixelInput is a type that vertex shader will return. Object of this type will be sent to fragment shader.
            PixelInput MainVs( VertexInput i )
            {
                // Create PixelInput object, initialize it with our processed vertex.
                // Between ProcessVertex() and FinalizeVertex() you can do any sorts of tricks
                // with your vertices.
                PixelInput o = ProcessVertex( i );

                // Return finalized vertex
                return FinalizeVertex( o );
            }
        }

Pixel Shader

This is a final stage of your shader. This is where we are going to manipulate the color and return it. For now, let's do a simple task: set up a simple, solid color and return it. This will make appear whole material with a certain color, unlit.

Adding the main function

Just like with vertex shader, create a new section. Give it a name PS and then add brackes. Inside the brackets, include another file - common/pixel.hlsl. It will import anything that fragment shader needs for rendering and compiling correctly.

After the include, create the main function of a fragment shader, float4 MainPs( PixelInput i ). float4 is a type that fragment shader returns, it must be exactly this type for now. PixelInput is the type that fragment shader receives from vertex shader. After typing this line, add : SV_Target0 at the end. It is a semantic that indicates that fragment shader returns a color. You probably won't ever need to touch this bit, just keep it as is.

Since we're doing a simple unlit shader, we're not utilizing anything from PixelInput data yet, but you will heavily utilize it in future articles when we move from unlit shader to a full, proper lit shader. 


PS
{
    #include "common/pixel.hlsl"

    float4 MainPs( PixelInput i ) : SV_Target0
    {
        // ...
    }
}

Quick note: "float" components

You're probably wondering, what the hell is a float4? If you know how Vector4 (or Vector3) works, then you can guess how floats work! Number after the float indicates how many components is in this vector. You can have floats types from float (single component vector) to float4 (four component vector).

Each component of a float vector can be accessed with suffixes .x, .y, .z and .w. Alternatively, you can access them with .r, .g, .b and .a, so .xyzw is an equivalent of .rgba. It doesn't matter which variant you are using, it's fine to use either.

float4 is often used to represent the color. As you may have probably guessed, RGBA is Red, Green, Blue and Alpha (aka transparency). This is why we are returning float4, because fragment shader needs to return the opacity too, even if it's not used by our shader yet.

Using suffixes, you can access one or several channels of a float vector at once, swizzle them or extract into a new variable with less components. int and uint can consists of multiple components too, just like floats. I'll provide some more information on floats, how to use suffixes and some other cool stuff in the next article where we'll add a texture to our unlit shader.

Adding color and returning it

This is very simple. This code can be much more compact, but just for the sake of learning we'll do it in a bit longer way. Create a new float3 variable, give it any name you like. For now, lets stick with "col". Initialize it by creating a new float3 with values set to 1, 0, 0.

Your like should look like this: float3 col = float3( 1, 0, 0 );

What does float3( 1, 0, 0 ) do? We just created a new three-component float with red (or X) channel set as 1, while others are 0. This will represent a pure red color. If you add 0.5 to green channel, then you'll get an orange color. If you add 1 to blue channel, you'll get a bright pink color. If you make them all have value 1, then you'll get white color. Have you ever seen the color picker window in Photoshop, with R, G, B input fields ranging from 0 to 255? It's the same here, except it's in 0-1 range!

Most values in shaders are represented within [0 - 1] range. Going beyond this range may cause visual mess. You can use saturate( float in ) to clamp your valie in 0-1 range if you expect that it'll be larger than that in realtime. Not sure if it is a good advice, but you probably want to write your math in a way where you don't have to rely on saturate() too much.

There's an exception with normals - they are operating in [-1 - 1] range.

Then we just have to return the color, but inside a float4 vector. Since our color just represents RGB without A component (opacity), shader won't compile because expected type is float4. Do float4( col, 1 ), and return it. 


PS
{
    #include "common/pixel.hlsl"

    float4 MainPs( PixelInput i ) : SV_Target0
    {
        float3 col = float3( 1, 0, 0 );
        float4 FinalColor = float4( col, 1 );

        return FinalColor;
    }
}

Combining or splitting floats is very easy, as you can see. You can initialize float4 by filling out all four values, or just add existing multi-component float and then fill out the value for missing component, like we did above. float3 col has RGB components already, and we're only missing component A, so we just created a new float4 with RGB components filled out by values from our variable col, and then added 1 for alpha component. 1 in alpha component will be usually interpreted as "fully opaque".

Before this article ends, lets simplify our code a little: 


PS
{
    #include "common/pixel.hlsl"

    float4 MainPs( PixelInput i ) : SV_Target0
    {
        return float4( float3( 1, 0, 0 ), 1 );
    }
}

Yeah! Instead of creating variables for each value seperately, we can just do this instead. Create a float4 vector, where we create a new float3 with pure red color, which automatically fills out RGB values, and then set float4's alpha component to 1, and then instantly return it. You can do it this way. But this is your code, and I trust that you know better what's more convenient for you. :-)

And here you go, you've got your first manually written unlit shader. Ignore these two random texture inputs: we'll get rid of them and add our own in the next article. Also, do you see all stuff on the left, like "rendering" and such? These are all the features included from our #import "common/features.hlsl" in FEATURES block, which means it worked!

EXTRA: My compiler is failing!

That's fine! You will probably deal with this a lot. All shader compiler logs are stored in your console. If something is failing to compile, it will dump all relevant information, and point to potential cause of the problem. It is also pretty good at catching typos and other basic mistakes.
