References

If you plan to continue learning shaders on your own, you must know where you can find references for s&box shaders. There's no Intellisense so I highly recommend looking up existing code.

  • s&box shaders location: Steam/steamapps/common/sbox/addons/base/assets/shaders
  • s&box Documentation on shaders. This documentation covers lots of s&box shader API, helper functions and other useful things.
  • s&box wiki. It isn't always up-to-date, but it never hurts to check it out.
  • Microsoft also provides a complete documentation for nearly everything what HLSL has. This probably won't be very useful for you as a beginner, but once you figure out the basics I recommend visiting it for any specific things.
I really encourage you to read source code for s&box shaders located in base addon, this may help figuring out some confusing parts about shaders. Do not hesitate to mess with your code: try changing values, math, functions and see what happens. Don't forget that you can visualize on screen nearly all data as colors. Don't be afraid to break shit, it's important to understand how data is represented in shaders and what every function does.

Default lighting model that s&box uses relies on PBR, aka Physically Based Rendering. This means that you should probably be familiar with default artist's PBR workflow, understanding why do you need such texture maps as albedo, roughness, normals, metalness, etc... This article on PBR by Marmoset staff seems to be pretty useful about the purpose of PBR and it's texture maps. And if you'd like to get extremely nerdy about it and understand the math behind it, "PBR Book" should be very helpful for you.

And another tip: take your time. If something isn't clicking in your head and you can't figure out some concept, try again later. Learning new things can fuck with your sanity sometimes, and sometimes the best solution is just "get some rest and come back to this later".

Do I need to understand programming?

Yeah. HLSL isn't a good language to start your programming adventures. If you have some experience with C#, you should feel comfortable with 99% of HLSL syntax. Just like most other languages, is is made of variables, classes, functions and statements. You can define custom functions and classes, and do any sorts of math.

If you're an artist looking to learn shaders and you have no programming experience, I highly recommend mess with C# first and then come back to this later. It is possible to learn HLSL without prior knowledge, but I imagine that it will be quite painful... but I am not going to stop you.

Software Recommendations?

It doesn't really matter, so pick any text editor you prefer. Some people use Visual Studio, some prefer to work in Visual Studio Code. Stuff like JetBrains Rider probably also works. If you're using VS Code, there are bunch of extensions that can improve HLSL code markdown to make navigating through code much easier.

If you decide to pick Visual Studio Code, I believe it will automatically recognize s&box shaders in ShaderLab format. This will properly highlight most of the code, although there will be some inaccuracies.

IntelliSense

As of now, there's no IntelliSense for HLSL code or s&box API. This may appear annoying at first, but there's a good thing about s&box API - it doesn't rely on internal functions and API too much. You won't need to remember hundreds of random macros, functions, or anything else. With enough practice, you'll feel comfortable typing code without having to constantly look up docs or some reference shaders, as long as you remember the HLSL syntax.
